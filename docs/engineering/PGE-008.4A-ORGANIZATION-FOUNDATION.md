# PGE-008.4A — Organization Foundation

## Status

Implementation review.

## Purpose

Establish the organization as Polaris' top-level tenant, data-isolation, and future authorization boundary.

## Scope

This increment introduces:

- persistent organization identity;
- unique normalized organization slugs;
- organization lifecycle status;
- create, list, and read APIs;
- canonical `organization.organization.created.v1` publication;
- organization and tenant identifiers on the creation event;
- conflict, validation, and not-found behavior;
- automated API and event-contract coverage.

## API

- `POST /api/v1/organizations`
- `GET /api/v1/organizations`
- `GET /api/v1/organizations/{organization_id}`

## Boundary

This increment does **not** claim to provide authentication, membership, roles, permissions, row-level authorization, federation, or production database migrations. Those belong to the following Identity and Access increments.

The existing legacy `Company` model remains unchanged. A later migration decision must define whether business companies become organizations, organization-owned entities, or both.

## Constitutional alignment

- **Builder Freedom:** organization identity is vendor-neutral.
- **Traceability:** creation publishes a canonical, idempotent event.
- **Human Authority:** no user access or role is inferred automatically.
- **Specification Before Implementation:** this document records the delivered contract and explicit limits.

## Acceptance criteria

- organizations receive stable unique identifiers;
- duplicate slugs return HTTP 409;
- malformed slugs return HTTP 422;
- unknown identifiers return HTTP 404;
- successful creation publishes `organization.organization.created.v1`;
- the event carries matching organization and tenant identifiers;
- the existing backend test suite remains green.

## Next increment

PGE-008.4B should introduce user identity and organization membership without embedding a single identity provider into the domain model.
