# PGE-008.4C — Authentication and Authorization Foundation

## Status

Implementation review.

## Purpose

Introduce a provider-neutral Security Kernel that turns an external credential into an organization-bound Polaris principal and applies deny-by-default permission decisions.

## Delivered capabilities

- `AuthenticationProvider` protocol independent of Microsoft, Google, or another vendor.
- Immutable `AuthenticatedPrincipal` carrying identity, organization, membership, role, permissions, provider, subject, and claims.
- Explicit permission vocabulary and role-to-permission policy.
- Active identity and active membership validation.
- Organization-bound principal resolution that denies cross-tenant access.
- Reusable FastAPI authentication and permission dependencies.
- Development-only signed, expiring local credentials.
- `POST /api/v1/auth/local/token` and `GET /api/v1/auth/me`.
- Canonical `identity.authentication.succeeded.v1` event publication.
- Automated coverage for successful authentication, invalid and expired credentials, tenant isolation, and permission policy.

## Request flow

1. Read bearer credential and `X-Polaris-Organization`.
2. Validate the credential through an authentication provider adapter.
3. Resolve the vendor-neutral Polaris identity.
4. Require an active membership in the requested organization.
5. Resolve permissions from the membership role.
6. Construct an immutable authenticated principal.
7. Deny unless the endpoint's required permission is present.

## Security decisions

- Authorization is permission-based; endpoints must not branch directly on role names.
- Missing permissions deny access.
- Unknown roles resolve to no permissions.
- A valid identity credential does not imply access to any organization.
- Local authentication is available only in development or test environments.
- The local signing secret must be overridden outside disposable development environments.

## Provider boundary

Microsoft Entra ID, Google Workspace, GitHub, OIDC, and SAML adapters can implement the provider protocol without changing identity, membership, principal, or authorization policy contracts.

## Explicit limits

This increment does not claim:

- production Microsoft Entra/OIDC discovery or key rotation;
- refresh tokens, browser sessions, logout revocation, or MFA;
- persistent authentication sessions;
- service accounts or machine identities;
- dynamic role administration or ABAC policy storage;
- complete protection of every legacy endpoint.

Endpoint migration to permission dependencies should proceed incrementally, starting with organization, identity, and connector management APIs.

## Canonical event

- `identity.authentication.succeeded.v1`

Authentication failure and authorization denial events require a non-sensitive audit publisher so invalid credentials are never copied into event payloads.

## Verification

- valid local credential creates a tenant-bound principal;
- missing, malformed, tampered, and expired credentials return HTTP 401;
- valid identity without membership returns HTTP 403;
- cross-organization access returns HTTP 403;
- owner permissions include organization management;
- viewer permissions exclude organization management;
- unknown roles receive no permissions;
- successful authentication publishes a tenant-aware canonical event;
- existing organization, identity, connector, and event behavior remain green.
