# PGE-008.5C — Scheduler Engine

## Status

Implementation review.

## Purpose

Introduce an in-process scheduler that decides which queued worker request becomes eligible for dispatch next.

## Delivered capabilities

- stable FIFO ordering for equal-priority work;
- explicit `Background`, `Low`, `Normal`, `High`, and `Critical` priorities;
- delayed availability through `availableAt`;
- retry scheduling with incremented attempt numbers;
- cancellation of queued work;
- a dispatcher boundary that hands due work to an injected executor;
- defensive copies for externally returned queue dates;
- automated coverage for ordering, delay, retry, cancellation, and dispatch behavior.

## Ordering rules

Eligible work is selected by:

1. highest priority;
2. earliest availability time;
3. original enqueue sequence.

Items whose `availableAt` is later than the scheduler clock are retained but cannot be dispatched.

## Architectural boundary

The scheduler owns queue order and eligibility. It does not execute worker business logic, persist queue state, or implement distributed coordination. The `Dispatcher` invokes an injected executor so Runtime integration can evolve without coupling the queue to Runtime internals.

## Explicit limits

This increment is intentionally in-process and memory-backed. It does not yet provide:

- durable or distributed queues;
- leases or visibility timeouts;
- concurrency limits;
- recurring schedules;
- backoff policy calculation;
- dead-letter queues;
- multi-node leader election.

Those capabilities require separate evidence-backed increments.

## Verification

- equal-priority work remains FIFO;
- higher-priority eligible work runs first;
- delayed work is not released early;
- retries preserve worker, payload, and priority while incrementing attempts;
- cancelled work is removed;
- dispatch delegates to the supplied executor;
- existing Runtime and lifecycle tests remain green.
